How this was built
System prompt design

The underlying Claude call is scoped narrowly to revision scheduling. It's instructed to: (1) always output strict JSON matching a fixed schema (no prose, no markdown fences) so the UI can render it directly into day cards; (2) weight extra time toward subjects the student flagged as weak; (3) respect hard constraints mentioned in free text or chat (e.g. "no Sundays"); (4) build in buffer/review days before the exam date rather than cramming until the last day; (5) never invent a syllabus topic list if none was given — instead ask for subject-level topics or work at subject granularity.

Edge cases handled
  • Exam date in the past or missing → the model is told to return an error field explaining what's wrong instead of a malformed schedule; the UI shows this in the error-state panel.
  • No subjects provided → same error pattern, prompts the user back to step 2.
  • Irrelevant / abusive input in the free-text or chat fields → the system prompt instructs Claude to ignore instructions embedded in student notes that try to change its role, and to stay focused only on producing a study plan.
  • Very short timelines (e.g. exam tomorrow) → the model is told to compress to high-yield review blocks rather than pretending a full revision cycle is possible.
Why these UI choices

A generic chat window hides the thing students actually want: a calendar they can scan at a glance. The horizontal day-card strip mimics a physical desk planner (torn-page dashed edges, a serif date numeral) so each day reads like a page you'd flip past. Weak subjects get a red-flagged block so revision time is visually prioritized, not just listed. The countdown strip at the top keeps exam urgency visible without needing to re-read the whole plan.

Extending this assistant
  • Tools: connect a calendar MCP server so "Generate" also writes events directly to the student's calendar, or a file-search tool to pull actual syllabus PDFs instead of pasted text.
  • Memory: persist the generated plan with window.storage so returning students see progress against their existing schedule instead of starting over, and can check off completed blocks.
  • Multi-step flows: add a daily check-in step where Claude re-plans the remaining days based on what was actually completed versus missed, using the same JSON schema.
  • Model swap: the fetch call targets claude-sonnet-4-6 — swap in a different model string if latency or cost needs change.
